# Заказ продуктов через T-Bank MCP

Этот скилл направляет агента через процесс заказа продуктов через T-Bank MCP (Город).
MCP обращается к реальным магазинам Т-Банка — реальные деньги, реальная доставка.
Всегда подтверждай с пользователем перед оплатой.

## Инструменты MCP (в порядке использования)

| Тул | Что делает |
|---|---|
| `grocery_stores()` | Список доступных магазинов (название, appId, pointId, время доставки, кешбэк). **Источник appId/pointId для всех остальных тулов.** |
| `grocery_search(query, app_id, point_id)` | Поиск товара по названию — возвращает товары с тегом `likely_raw`. **app_id/point_id обязательны.** |
| `grocery_plan_order(ingredients, app_id, point_id)` | Умный планировщик: ищет каждый ингредиент (сначала custom_ordered, потом глобально) → план. **app_id/point_id обязательны.** |
| `grocery_add_to_cart(items, app_id, point_id)` | Добавить товары в корзину (автоматически берёт адрес доставки с details). **app_id/point_id обязательны.** |
| `grocery_cart(app_id, point_id)` | Получить содержимое корзины (проверить перед оплатой). **Те же app_id/point_id что в add_to_cart.** |
| `grocery_checkout(app_id, point_id)` | Полный чекаут: web-корзина → доставка → заказ → оплата. РЕАЛЬНЫЕ ДЕНЬГИ. **Те же app_id/pointId.** |

> ⚠️ **app_id и point_id обязательны во всех grocery-тулах кроме `grocery_stores`.** Без них тул
> вернёт `NO_STORE_CONTEXT`. Дефолтный магазин вслепую НЕ подставляется — иначе корзины
> add_to_cart / cart / checkout разъедутся по разным магазинам. Каждый ответ начинается с
> `[store appId=… pointId=…]` — сверяй, что магазин тот же. Если увидишь `CART_CONTEXT_MISMATCH` —
> контекст рассинхронизирован, перечитай `grocery_stores()` и передай согласованные appId/pointId.

## Флоу (всегда в таком порядке)

### Шаг 1: Определить что заказывать

**Если пользователь называет блюдо** (борщ, оливье, плов и т.д.):

1. Попробуй определить ингредиенты из своих знаний (LLM). Для популярных блюд
   (борщ, оливье, плов, паста, блины, котлеты, супы) — этого достаточно.

2. **Если блюдо незнакомое или необычное** — используй WebSearch:
   ```
   WebSearch("рецепт {название блюда} ингредиенты список")
   ```
   Найди рецепт → извлеки список ингредиентов из результата.
   Если результатов несколько — возьми первый подходящий (eda.ru, povarenok.ru,
   championat.com, gastronom.ru — авторитетные кулинарные сайты).

3. Покажи пользователю полный список. Если пользователь что-то назвал, а ты по
   рецепту знаешь что ещё нужно — **скажи прямо**:
   «Для борща нужно: свёкла, говядина, капуста, морковь, лук, чеснок,
   картофель, томатная паста. Ты не назвал свёклу и морковь — они основные!
   Добавить их?»
   Дождись подтверждения финального списка.

**Если пользователь перечисляет ингредиенты сам** (не называет блюдо):

1. Сравни с известными рецептами — не пропустил ли пользователь что-то важное.
   Примеры типичных забытых ингредиентов:
   - Борщ: часто забывают свёклу, томатную пасту, уксус/лимон (для цвета)
   - Оливье: часто забывают морковь, горошек, огурцы, майонез
   - Плов: часто забывают чеснок, зиру, барбарис, нут
   - Картофельное пюре: часто забывают молоко/масло
   - Паста: часто забывают пармезан, оливковое масло
   - Блины: часто забывают молоко, яйца, масло

2. Если не уверен в полноте списка (блюдо не знакомо) — WebSearch:
   ```
   WebSearch("{названные ингредиенты} что за блюдо рецепт")
   ```
   Определи блюдо → найди полный рецепт → сравни с тем что пользователь назвал.

3. Если чего-то не хватает — **не просто предложи, а укажи прямо**:
   «Вообще-то в рецепт ещё входят огурцы и морковка! Добавить?»
   Будь как знающий друг — если забыт основной ингредиент (свёкла для борща),
   скажи об этом с акцентом: «Свёкла — это ОСНОВНОЙ ингредиент борща!»

4. Если пользователь сказал «закажи продукты для борща, но только говядину и картошку»
   (сознательно урезал) — НЕ настаивай, просто подтверди: «Заказываю только говядину
   и картошку. Остальное пропускаю. Верно?»

5. Дождись подтверждения финального списка.

### Шаг 2: Выбрать магазин

1. Вызови `grocery_stores()` — увидеть доступные магазины.
2. Покажи пользователю топ 3-5: название, время доставки, минимальная сумма заказа, кешбэк %, но только если просит.
3. **Приоритет**: предпочитай магазины где пользователь уже заказывал (если у магазина есть категория `custom_ordered` — там есть история заказов).
4. Иначе выбери с самой быстрой доставкой + низкой минимальной суммой.
5. Запомни выбранные `appId` и `pointId` — передавай их во все последующие вызовы.

### Шаг 3: Поиск ингредиентов

**a. Вызови `grocery_plan_order(ingredients_list, app_id, point_id)`** — один вызов ищет всё:
- Сначала ищет в `custom_ordered` («Вы заказывали») — товары которые пользователь покупал раньше.
- Потом глобальный поиск (`search/fulltext` с `inStockFilter` — только в наличии).
- Возвращает: `{items, total_sum, missing}`.

**b. Если в плане есть missing-позиции** — ищи каждый вручную:
```
grocery_search("свёкла", app_id, point_id)
```
- Результаты отсортированы: `likely_raw: true` сначала, потом готовая еда.
- **Предпочитай `likely_raw: true`** (сырые ингредиенты) если пользователь не просил готовую еду.
- Если сырого нет (например «свёкла» только «Свёкла с майонезом»):
  - Скажи пользователю: «Сырой свёклы нет в этом магазине. Нашёл: [список]. Заменить или заказать из другого магазина?»

### Шаг 4: Стратегии поиска (уточни у пользователя или примени дефолтную)

Пользователь может указать стратегию. Если не указал — по умолчанию «лучшее совпадение из `custom_ordered`»:

| Стратегия | Как применить |
|---|---|
| **Самое дешёвое** | `grocery_search` возвращает до 10 результатов с полем `price`. Отсортируй по цене (возрастание), выбери самый дешёвый `likely_raw` товар. Если цена за разный вес — сравнивай цену за кг (раздели `price` на `weight`). |
| **Самое дорогое / премиум** | То же, но сортируй по убыванию. Выбери самое дорогое `likely_raw`. Подходит когда пользователь хочет лучшее качество (часто дороже = лучше). |
| **Цена за кг** | Внимание: вес товаров разный! «Картофель 2.8кг за 160₽» = 57₽/кг, а «Картофель 1кг за 120₽» = 120₽/кг. Считай цену за единицу веса (поле `weight` в результате). Всегда сравнивай по цене за кг/литр, а не по итоговой цене. |
| **Минимальное КБЖУ** | КБЖУ нет в API (TODO). Используй знания LLM: куриная грудка < свинина < говядина по калориям. Выбери самый низкокалорийный сырой вариант. |
| **Из любимого / ранее заказанного** | `custom_ordered` — ближайшее к «любимому». `grocery_plan_order` уже проверяет это первым. |
| **Лучшее качество** | Предпочитай бренды магазина (««Просто Азбука»», ««Наша Ферма»») — обычно свежее. |
| **Конкретный бренд** | Ищи по названию бренда: `grocery_search("Borges оливковое масло")`. |
| **Бюджет на всё блюдо** | Если пользователь назвал бюджет (например «до 2000₽ на борщ»): посчитай сумму выбранных товаров. Если превысил — замени дорогие позиции на более дешёвые варианты того же ингредиента. Покажи сравнение: «Говядина Orenbeef 2473₽ vs Говядина обычная 1200₽ — разница 1273₽. Меняем?» |

### Шаг 5: Обработка отсутствующих товаров (кросс-магазин)

Если часть ингредиентов не найдена в выбранном магазине:

1. **Сообщи пользователю**: «Не нашёл в {магазин}: свёкла, томатная паста. Варианты: (a) заказать из другого магазина, (b) заменить, (c) пропустить.»
2. Если пользователь хочет заказать из другого магазина:
   - Найди отсутствующие товары в другом магазине (другой appId/pointId).
   - **Важно**: у каждого магазина ОТДЕЛЬНАЯ корзина. Нужно сделать отдельный `grocery_add_to_cart` + `grocery_checkout` для второго магазина.
   - Покажи обе корзины пользователю перед оплатой.
3. Если пользователь хочет замену:
   - Поищи альтернативы (например «свёкла отварная» если сырой нет).
4. Если хочет пропустить — просто убери позицию из заказа.

### Шаг 6: Сборка корзины

1. Вызови `grocery_add_to_cart(items, app_id, point_id)` с выбранными ID товаров:
   ```json
   [{"id": "248429", "count": 1}, {"id": "368285", "count": 1}, ...]
   ```
2. Вызови `grocery_cart(app_id, point_id)` — с **теми же** app_id/point_id — чтобы проверить содержимое корзины.

### Шаг 7: Показать корзину пользователю — ОБЯЗАТЕЛЬНО перед оплатой

**Всегда** показывай полную корзину перед `grocery_checkout()`:

```
Корзина (Азбука Вкуса):
  1. Лук репчатый — 89 ₽ [СЫРОЙ]
  2. Говядина бескостная — 2473 ₽ [СЫРОЙ]
  3. Картофель — 160 ₽ [СЫРОЙ]
  4. Свёкла с майонезом — 354 ₽ [ГОТОВАЯ — сырой нет]
  ...
  Итого: 4165 ₽, доставка бесплатная, ~40 мин

Подтверждаешь заказ? (да / нет / изменить)
```

**НЕ вызывай `grocery_checkout()` пока пользователь не подтвердит.** Если сказал «изменить»:
- Убери/добавь товары, смени магазин — потом покажи корзину снова.
- Только после явного «да» или «заказывай» или «оплачивай» — переходи к оплате.

### Шаг 8: Оформление + оплата

1. Вызови `grocery_checkout(app_id, point_id)` — с **теми же** app_id/pointId что в корзине. Это делает всё:
   - Синхронизация web-корзины (portalSID + sessionID cookies)
   - GET реальная сумма из web-корзины (весовые товары могут отличаться от мобильной суммы)
   - POST доставки (инициализация слотов доставки)
   - POST order/create
   - POST payment_gate_pay (с `currencyCode=643`, сразу — заказы авто-отменяются если не оплатить быстро)
   - ⚠️ Web-flow чекаута (`src/checkout.py`) пока надёжно работает для `appId=578` (Азбука Вкуса).
     Для другого магазина чекаут может не пройти — тогда скажи пользователю и предложи магазин 578
     либо ожидание починки web-flow (Phase 3).
2. Сообщи результат: ID заказа, ID платежа, итоговая сумма.
3. Если оплата не прошла (500/ошибка) — скажи пользователю, предложи повторить или оплатить в приложении.

### Шаг 9: Отчёт

После успешного заказа:
```
✅ Заказ оформлен!
  Заказ: 476748521145
  Платёж: 127301539918 (SUCCESS)
  Сумма: 884 ₽
  Магазин: Азбука Вкуса
  Доставка: ~40 мин

Не заказано: свёкла (нет в этом магазине)
```

Если были товары из второго магазина — укажи оба заказа.

## Ключевые правила

1. **Один магазин на корзину + явный контекст** — нельзя смешивать товары из разных магазинов в одном заказе. Передавай `app_id`/`point_id` (из `grocery_stores()`) в **каждый** grocery-тул; без них — `NO_STORE_CONTEXT`. Цепочка `add_to_cart → cart → checkout` должна идти с **одинаковыми** app_id/pointId. Каждый ответ начинается с `[store appId=… pointId=…]` — сверяй его.
2. **Тег `likely_raw`** — предпочитай сырые ингредиенты (тег=true) если пользователь не сказал иное.
3. **Без подтверждения — без оплаты** — всегда показывай корзину первым.
4. **Плати сразу после заказа** — заказы авто-отменяются если не оплатить за ~5 минут.
5. **`custom_ordered` = приоритет** — товары которые пользователь покупал раньше, предпочитать.
6. **`currencyCode=643`** — в теле платежа должно быть это (обрабатывается в `grocery_checkout()`).
7. **Нет в наличии = блок заказа** (code 211). Убери такие товары перед заказом.
8. **КБЖУ нет в API** — используй знания LLM для оценки калорий/белков/жиров/углеводов.
