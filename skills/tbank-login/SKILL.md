# Логин и управление сессией T-Bank MCP

## Инструменты

| Тул | Что делает |
|---|---|
| `login(phone)` | Начинает SSO-логин. Отправляет SMS OTP. Если пароль задан в TBANK_PASSWORD env — автоматически подставляет. |
| `confirm_otp(otp)` | Отправляет SMS-код. Пароль (если банк просит) автоматически берётся из TBANK_PASSWORD env. |
| `refresh_session()` | Тихий re-login (без OTP/пароля). Использует SSO_SESSION + fingerprint. |
| `session_status()` | Проверить жива ли сессия. |
| `keepalive()` | Пинг — продлевает сессию (~2ч TTL). |

## ⚠️ БЕЗОПАСНОСТЬ: пароль и PIN никогда не попадают к агенту

Пароль аккаунта и PIN — это секреты. Они **НЕ** передаются агенту (LLM).
Вместо этого:

**Способ 1 (рекомендуемый) — env переменная:**
```bash
export TBANK_PASSWORD="ваш_пароль"
# теперь login() + confirm_otp() используют пароль автоматически
```

**Способ 2 — локальный CLI (вне агента):**
```bash
python login_cli.py "+79991234567"
# вводит OTP и пароль локально, сохраняет session.json
```

Агент (LLM) видит только phone и OTP (одноразовый код — не секрет).

## Когда что вызывать

### Первый логин (session.json нет или сессия полностью истекла)

```
пользователь: «залогинься» / «хочу посмотреть счета»
    ↓
1. login("+79991234567")
    → SMS отправлена. Возвращает: "step: otp" (или "password" или "pin")
    ↓
2. Пользователь получает SMS → сообщает код
    ↓
3. confirm_otp("1234")
    → Если вернёт "step: password" → следующий шаг
    → Если вернёт session → готово (редкий случай, обычно без пароля)
    ↓
4. Банк просит пароль (первый логин на новом устройстве)
    Пользователь сообщает пароль
    ↓
5. confirm_password("ваш_пароль")
    → session minted! session.json сохранён.
    ↓
6. list_accounts() → 12 счетов. Готово.
```

**Важно:** `login()` возвращает КАКОЙ ШАГ следующий. Агент вызывает
соответствующий `confirm_*`. Не знает заранее — узнаёт из ответа login().

### Сессия истекла (access_token ~2ч, но SSO_SESSION жив)

```
тул вернул: SESSION EXPIRED / INSUFFICIENT_PRIVILEGES
    ↓
1. refresh_session()
    → тихий re-login (без OTP, без пароля). Использует SSO_SESSION + fingerprint.
    → новый access_token + sessionid. session.json обновлён.
    ↓
2. Повторить тул, который упал.
```

### Сессия полностью мертва (SSO_SESSION истёк/отозван)

```
refresh_session() тоже падает (invalid_grant)
    ↓
Нужен ПОЛНЫЙ логин заново: login(phone) → OTP → password
    ↓
Сказать пользователю: «Сессия истекла полностью. Нужно залогиниться заново.
Назови номер телефона.»
```

### Проактивное обновление (не ждать истечения)

MCP сам вызывает `refresh_session()` перед каждым тулом (`ensure_fresh`).
Агенту не нужно делать это вручную — но если тул упал с SESSION EXPIRED:
вызови `refresh_session()` → повтори тул.

## Ключевые правила

1. **login() → узнаёшь шаг → confirm_*** — не угадывай какой confirm нужен,
   читай ответ login() (otp / password / pin).

2. **Токен чейнится** — каждый шаг возвращает новый токен для следующего.
   Не нужно запоминать — MCP делает это внутри (confirm_step).

3. **Пароль нужен только при первом логине на новом устройстве.**
   При повторных логинах (SSO_SESSION жив) — refresh_session() без пароля.

4. **PIN ≠ пароль.** PIN — 4-значный код приложения (re-auth).
   Пароль — пароль аккаунта Т-Банка (первый логин).

5. **session.json не коммитить!** — там токены. Он gitignored, но на диске нужен.

6. **Не логиниться без необходимости** — если session_status() = OK,
   не вызывай login. Используй refresh_session() если истёк access_token.

7. **Rate limit на SMS** — если login(phone) вызывался多次, банк может
   не отправить SMS. Жди ~10 мин между попытками.

8. **PIN lockout** — 3 неверных PIN → блокировка. Если confirm_pin упал с
   "wrong_pin" — НЕ повторяй слепо. Спроси пользователя.
